# RevitCli v2.1 — "Configuration Confidence"

> Period: starts immediately after v2.0 §7 stack merges.
> Theme: deepen ALREADY-DELIVERED capabilities so operators trust them
> in production. v2.1 is a **maintenance + verification** version, not
> a new-feature version.
> Workflow: same `superpowers` cadence as the §7 stack
> (brainstorm → spec → plan → execute), but with smaller per-PR scope.
> Next roadmap: [`roadmap-2026q4-v4.md`](./roadmap-2026q4-v4.md).

---

## 0. Why v2.1 and not v3.0

The v1.x → v2.0 ramp built six milestones in a row (§7 of the v1.5–v2.0
roadmap). v2.0 §7 closes the dashboard surface; the project now has
real BIMOps capability ON PAPER. v2.1's job is to **convert that to
trustworthy production capability** without expanding scope.

Concretely:

- v2.0 ships multi-Revit-version SUPPORT but only one version
  (`Revit 2026`) is exercised by `smoke-revit2026.ps1`. The README
  claims 2024 / 2025 / 2026; nobody knows whether the claims hold.
- v1.9 ships profile lint (`profile validate`) and v2.0 ships the
  governance surface, but **runtime errors in publish pipelines are
  still a thing**: missing presets, wrong format strings, empty
  selectors. Operators only find out after `publish` is mid-flight.
- The `journal.jsonl` audit trail is append-only but unsigned. For
  legal / regulated industries (hospitals, government, public-sector
  AEC) that's not enough — they want tamper-evident logs.

v2.1 ships the three "confidence" missing pieces. None invents new
verbs; each deepens an existing one.

### Step 1 — `profile simulate <pipeline>` ✅

Closes the gap between schema lint and runtime errors.

`profile validate` checks that the YAML parses. `profile simulate
<name>` actually walks a publish pipeline as if it were going to run,
without contacting Revit:

```bash
$ revitcli profile simulate client-A
Pipeline 'client-A' (from /repo/.revitcli.yml)
==============================================
  Precheck: 'default'  failOn=error
    rules: naming, room-bounds, required-parameter:doors.Mark
  Presets (3):
    - dwg-bidding   format=dwg, sheets=[A1*]
    - pdf-archive   format=pdf, sheets=[ALL]
    - ifc-export    format=ifc, sheets=[ALL]
  Incremental: yes  baseline=.revitcli/last-publish.json  sinceMode=content
  Webhook: https://hooks.example.com/revitcli

Findings (2):
  [info] preset-all-sheets: Preset 'pdf-archive' uses 'sheets: ALL'. Confirm intentional.
  [info] preset-all-sheets: Preset 'ifc-export' uses 'sheets: ALL'. Confirm intentional.
```

What the simulator catches that `validate` doesn't:

- **Reference completeness** — every preset name and precheck name
  the pipeline mentions actually resolves in `exports.<name>` /
  `checks.<name>`. (Catches typos that schema lint can't see.)
- **Preset format sanity** — `dwg|pdf|ifc` against the runtime's
  expectations.
- **Selector surface** — `sheets: ALL` (info), empty selectors
  (warning), missing format (error).
- **Precheck signal** — aggregates `auditRules` + `requiredParameters`
  - `naming` into one rule list; flags an empty precheck.

Exit-code policy mirrors `audit` and `family validate`:
`--fail-on error` (default) blocks CI on any error finding;
`--fail-on warning` ratchets to warning; `--fail-on info` makes the
gate strictest (every finding fails). `--output table | json`.

### Step 2 — Multi-Revit-version smoke matrix ✅

The README claims `Revit 2024 / 2025 / 2026`. Today only 2026 is
exercised by `scripts/smoke-revit2026.ps1`. Step 2 ships the
multi-version smoke scaffolding:

- `scripts/smoke-revit.ps1 -Version 2024|2025|2026` — single
  parameterized smoke that validates the requested Revit install,
  add-in manifest, live server, and canonical query → set → restore loop.
- `revitcli doctor --check-version 2024|2025|2026` — version-aware
  prerequisite and live-connection diagnostics.
- `docs/ci/smoke-matrix-template.yml` — Windows self-hosted runners
  labelled `revit-2024` / `revit-2025` / `revit-2026`.
- `docs/revit-version-compatibility.md` — compatibility notes and the
  per-version surfaces that still need live fleet confirmation.

Full proof still requires three Windows + Revit machines; the shipped
matrix is the operator handoff for that fleet validation.

### Step 3 — Journal signing + verify ✅

`.revitcli/journal.jsonl` is append-only but unsigned. Step 3 adds:

- `revitcli journal sign` — produces a SHA256-chained Merkle-style
  signature file `.revitcli/journal.jsonl.sig` covering all
  entries up to a given timestamp.
- `revitcli journal verify` — re-walks the journal and the signature,
  reports any tampering (entry insertion, deletion, modification).
- `--key <path>` — custom local HMAC key path; default is
  `.revitcli/journal.key`, created on first sign.
- `--output json` — CI-friendly machine-readable result.

The first shipped version is a local HMAC trust model. RFC-3161 /
Sigstore-style public timestamping remains a future deployment
integration, not part of v2.1 core.

## 2. Out of scope for v2.1

- **Revit Family Editor** writes (deferred to v3.x per v1.8 follow-up
  notes — needs Family Editor API context that's qualitatively
  harder than `Document.Delete`).
- **Cross-document publish** (Revit worksharing). Out of scope across
  the entire v1.x / v2.x range.
- **MCP feature expansion**. MCP is no longer a roadmap direction for
  RevitCli; future work stays focused on architects using the terminal.

## 3. Backwards compatibility contract

Same rules as the v1.5–v2.0 roadmap §9:

- `IRevitOperations` is additive only.
- `.revitcli.yml` profile schema: new fields land with defaults; no
  v1.0–v2.0 profile becomes invalid.
- `journal.jsonl` shape: `transport` / `outcome` / `action` etc.
  fields stay where they are; new entries can carry additional
  fields without breaking older readers.

## 4. Non-goals (rephrased from v1.5–v2.0 §9)

- No SaaS layer. Federation across orgs is a **deploy** concern, not a
  product feature.
- No bundled LLM and no MCP expansion. External terminal operators such
  as Codex CLI may call RevitCli through normal shell commands; RevitCli
  itself should stand on commands, files, receipts, and repeatable
  workflows.
